## Why

The live GitHub Pages site for the repository is currently displaying unstyled, broken content because GitHub Pages is defaulting to the built-in Jekyll generator instead of serving the VitePress build artifacts. Despite having a GitHub Actions workflow (`deploy.yml`) configured to build and deploy VitePress, the repository settings are likely still set to "Deploy from a branch" (or the branch contains a Jekyll-compatible structure that GitHub intercepts). To ensure VitePress is strictly served and Jekyll is bypassed even if branch deployment is used, we need to add a `.nojekyll` file to the VitePress public directory.

## What Changes

- Add a `.nojekyll` file to the `docs/public` directory (or ensure it's generated by VitePress) to explicitly disable GitHub Pages' default Jekyll processing.
- Ensure the build output includes this `.nojekyll` file so that when GitHub Pages serves the `dist` directory, it bypasses Jekyll and serves the VitePress HTML/CSS/JS exactly as built.

## Capabilities

### New Capabilities
- `github-pages-deployment`: Ensures strict VitePress deployment bypassing Jekyll.

### Modified Capabilities
- None

## Impact

- **UI/UX**: The live site will correctly load VitePress styles, scripts, and routing instead of rendering broken Jekyll Markdown pages.
- **Infrastructure**: The deployment process on GitHub Pages will strictly serve the static files generated by VitePress.
